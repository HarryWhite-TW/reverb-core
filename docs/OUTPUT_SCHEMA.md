# Reverb Output Schema

## 1. Purpose

This document describes the current output shapes for Reverb.

It distinguishes between:

- in-memory Python `ProcessingResult`
- CLI JSON output

## 2. In-Memory ProcessingResult Shape

Current object shape from `elysia_core.contracts.ProcessingResult`:

| Field | Type / Shape | Meaning | Notes |
| --- | --- | --- | --- |
| `correlation_id` | `str` | Per-call trace identifier. | Generated by the pipeline. |
| `original_input` | `Any` | Original input value. | May be a string or another type such as `None`. |
| `processed_text` | `str` | Final processed text. | May be normalized text or fallback text. |
| `is_valid` | `bool` | Whether the result is valid. | `False` for fallback and type guard cases. |
| `events` | `list[StepEvent]` | Processing events. | Full in-memory event objects. |
| `errors` | `list[ErrorItem]` | Processing errors. | Full in-memory error objects. |

## 3. In-Memory StepEvent Shape

Current object shape from `elysia_core.contracts.StepEvent`:

| Field | Type / Shape | Meaning | Notes |
| --- | --- | --- | --- |
| `name` | `str` | Event / step name. | Examples include `strip_spaces`, `fallback_if_empty`. |
| `severity` | `Literal["info", "warn", "error"]` | Event severity. | Normal steps generally use `info`; guardrails may use `warn`. |
| `changed` | `bool` | Whether the step changed the text or processing state. | Used by normal step events and guardrail events. |
| `note` | `str` | Human-readable event note. | Present in the in-memory object. |
| `before` | `Optional[str]` | Value before the step. | May be `None`; type guard may store a non-string runtime value. |
| `after` | `Optional[str]` | Value after the step. | May be `None`. |

## 4. In-Memory ErrorItem Shape

Current object shape from `elysia_core.contracts.ErrorItem`:

| Field | Type / Shape | Meaning | Notes |
| --- | --- | --- | --- |
| `code` | `str` | Machine-readable error code. | Examples include `EMPTY_INPUT`, `UNEXPECTED_TYPE`. |
| `message` | `str` | Human-readable error message. | Current messages may contain non-ASCII text. |
| `step` | `str` | Step that emitted the error. | Examples include `type_guard`, `fallback_if_empty`. |
| `severity` | `Literal["info", "warn", "error"]` | Error severity. | Defaults to `error`; current guardrail errors may use `warn`. |

## 5. CLI JSON Output Shape

Current CLI JSON shape from `elysia_core.cli`:

| Field | Type / Shape | Meaning | Notes |
| --- | --- | --- | --- |
| `processed_text` | `str` | Final processed text. | JSON emits the actual text value. |
| `is_valid` | `bool` | Whether the result is valid. | Mirrors `ProcessingResult.is_valid`. |
| `errors` | `list[str]` | Error code strings. | CLI JSON emits `ErrorItem.code`, not full `ErrorItem` objects. |
| `events` | `list[object]` | Event summaries. | Each event includes only `name`, `changed`, and `severity`. |
| `correlation_id` | `str` | Per-call trace identifier. | Mirrors `ProcessingResult.correlation_id`. |

CLI JSON does not currently emit:

- `original_input`
- full `ErrorItem.message`
- full `ErrorItem.step`
- full `StepEvent.note`
- full `StepEvent.before`
- full `StepEvent.after`

## 6. Representative CLI JSON Cases

Commands:

```powershell
python -m elysia_core.cli --json "What!!??"
python -m elysia_core.cli --json "   "
```

Expected output descriptions:

- valid processed_text should equal `What\uFF01\uFF1F`
- fallback processed_text should equal `\u2026`

## 7. Error Codes Currently Demonstrated

Currently demonstrated error codes:

- `EMPTY_INPUT`: emitted when empty or whitespace-only input triggers fallback.
- `UNEXPECTED_TYPE`: emitted when non-string input is passed to `preprocess_input()`.

The CLI cannot demonstrate `UNEXPECTED_TYPE` through normal command-line arguments because argv values are strings.

## 8. Event Names Currently Demonstrated

Currently demonstrated event names:

- `type_guard`
- `strip_spaces`
- `trim_edges`
- `collapse_spaces`
- `fallback_if_empty`
- `symbol_cleaner`

## 9. Correlation ID

`correlation_id` is generated per call.

Based on current tests and implementation, it is expected to be a 32-character lowercase hex string.

It should not be treated as deterministic for the same input.

## 10. Stability Notes

v0.7/v0.8 currently documents observed behavior.

Reverb is not yet SDK-complete.

Future versions may formalize schema compatibility.
