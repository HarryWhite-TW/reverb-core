# Reverb Python Library API Reference

## 1. Purpose

This document describes the current Python library API surface for Reverb. It is not HTTP, REST, or web API documentation.

Reverb is currently a deterministic input guardrail / preprocessing core. This is not a completed SDK reference yet.

## 2. Current Public Entry Point

Current public entry point:

```python
elysia_core.input.preprocess.preprocess_input
```

`preprocess_input(text)` runs the deterministic preprocessing pipeline for one input value.

Input expectation:

- Preferred input is `str`.
- Non-string input is handled by the current type guard and returns a structured invalid result.

Return type:

- `ProcessingResult`

## 3. Basic Library Usage

```python
from elysia_core.input.preprocess import preprocess_input

result = preprocess_input("What!!??")

print(result.processed_text.encode("unicode_escape").decode())
print(result.is_valid)
print(result.errors)
print(result.events)
```

Expected output description:

- processed_text should equal `What\uFF01\uFF1F` when escaped as `unicode_escape`

## 4. ProcessingResult Overview

Current fields from `elysia_core.contracts.ProcessingResult`:

| Field | Meaning |
| --- | --- |
| `correlation_id` | Per-call trace identifier generated by the pipeline. |
| `original_input` | The original value passed to `preprocess_input()`. |
| `processed_text` | Final processed text after preprocessing or fallback. |
| `is_valid` | Boolean validity flag for the processed result. |
| `events` | List of `StepEvent` objects emitted during processing. |
| `errors` | List of `ErrorItem` objects emitted during processing. |

There is no `reasons` field in the current `ProcessingResult` contract.

## 5. StepEvent Overview

Current fields from `elysia_core.contracts.StepEvent`:

| Field | Meaning |
| --- | --- |
| `name` | Processing step name, such as `strip_spaces` or `symbol_cleaner`. |
| `severity` | Step severity: `info`, `warn`, or `error`. |
| `changed` | Whether the step changed the text or processing state. |
| `note` | Human-readable note for the event. |
| `before` | Optional value before the step. |
| `after` | Optional value after the step. |

## 6. ErrorItem Overview

Current fields from `elysia_core.contracts.ErrorItem`:

| Field | Meaning |
| --- | --- |
| `code` | Machine-readable error code. |
| `message` | Human-readable error message. |
| `step` | Step that produced the error. |
| `severity` | Error severity, defaulting to `error`; current guardrail warnings may use `warn`. |

## 7. Representative Cases

| Case | Input | processed_text escape | is_valid | Error codes | Event names |
| --- | --- | --- | --- | --- | --- |
| valid symbol normalization | `"What!!??"` | `What\uFF01\uFF1F` | `True` | `[]` | `strip_spaces`, `trim_edges`, `collapse_spaces`, `symbol_cleaner` |
| fallback / empty input | `"   "` | `\u2026` | `False` | `EMPTY_INPUT` | `strip_spaces`, `trim_edges`, `collapse_spaces`, `fallback_if_empty` |
| unexpected type | `None` | `\u2026` | `False` | `UNEXPECTED_TYPE` | `type_guard` |

## 8. CLI Relationship

The CLI is currently a demo / validation adapter.

It can be run through:

```powershell
python -m elysia_core.cli
```

CLI JSON output may serialize errors/events differently than the in-memory Python objects:

- CLI JSON `errors` currently contains error code strings.
- CLI JSON `events` currently contains only `name`, `changed`, and `severity`.
- Direct library use should rely on the full `ProcessingResult` object.
